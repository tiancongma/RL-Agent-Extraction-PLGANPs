# Results Writer Audit (2026-04-15)

This audit was read-only. I searched `src/`, `tools/`, and `scripts/` for writers that create or populate `data/results/` paths, then checked each writer against the governed run naming and lineage contract in `AGENTS.md`, `project/ACTIVE_PIPELINE_RUNBOOK.md`, `project/ACTIVE_DATA_SOURCE_CONTRACT.md`, `project/FILE_NAMING_AND_VERSIONING.md`, and `docs/maintained_script_surface.tsv`. The main bypass pattern is direct construction of `data/results/{run_id}/...` or fixed top-level `data/results/...` outputs without `resolve_results_write_target(...)`, without governed `RUN_CONTEXT.md` emission, or with recency-based run discovery.

## Notes

- I did not find additional `data/results/` writer violations under `scripts/`; the only Python file there is `scripts/repo_governance_check.py`, which does not write into `data/results/`.
- No `tools/` directory was present in this workspace during the audit.
- I did not flag scripts that write only under an already resolved governed run and validate the path through `resolve_results_write_target(...)` or equivalent governed source resolution.
- The strongest recurring repair target is straightforward: any script that can create or populate a new `data/results/...` location should resolve its output root through `src/utils/run_id.py` and emit reproducibility metadata including `RUN_CONTEXT.md`.
