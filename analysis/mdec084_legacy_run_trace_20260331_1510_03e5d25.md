## Why Legacy Naming Was Still Used

Two repository facts explain the legacy path:

1. the caller explicitly passed a legacy `--run-id`
2. the earlier MDEC084 utility pass had not yet enabled v2 writes by default

Supporting evidence:

- `analysis/mdec084_utility_migration_validation.md` explicitly says:
  - `default legacy generation remains unchanged`
  - `build_run_id(...) still emits legacy run_YYYYMMDD_HHMM_<short_hash>_<suffix>`
  - `no writer was switched to emit the v2 layout by default`

So this legacy run was not an unexplained regression. It was the expected
pre-rollout behavior of the maintained Stage2 writer.

## Expectedness Under Pre-Rollout Behavior

- expected under pre-rollout behavior: `yes`

Reason:

- MDEC084 had already defined the future naming contract and helper functions,
  but real maintained write entrypoints still defaulted to legacy naming or
  required explicit legacy `--run-id` values.
- The governed Stage2 entrypoint had not yet been switched to default v2
  bucket/child allocation.

## Outcome Of This Rollout Pass

This rollout pass changes the default-writer behavior for future runs only:

- maintained write entrypoints now default to MDEC084 v2 bucket/child naming
- explicit legacy `--run-id` remains compatibility-only
- historical `run_*` surfaces, including this March 31 run, remain unchanged
