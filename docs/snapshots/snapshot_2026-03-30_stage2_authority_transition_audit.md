# Section 1: Executive conclusion

The repo evidence supports an incremental, three-step transition rather than a single clean switch. The earliest **role narrowing** happened on `2026-03-06`, when the architecture contract explicitly split LLM semantic responsibility from deterministic normalization and export work in [docs/methods/llm_deterministic_responsibility_contract_2026-03-06.md](../methods/llm_deterministic_responsibility_contract_2026-03-06.md) and the same rule was echoed in [project/4_DECISIONS_LOG.md](../../project/4_DECISIONS_LOG.md). The earliest clear **practical drift** happened by `2026-03-13`, when deterministic numbered-DOE enumeration was made primary inside the still-active legacy extractor path, reducing the LLM from row counter to judge in that subproblem. The **formal authority switch** was not on `2026-03-25`: that date approved the semantic replacement direction and compatibility bridge while explicitly saying the active runtime was still the wide-row LLM extractor. The best-supported formal switch happened in commit `75c148e` on `2026-03-26`, which rewrote the active pipeline docs, README, architecture, runbook, script map, and memory (`MDEC076`) to make `emit_semantic_objects_from_cleaned_papers_v1.py` authoritative and `build_stage2_compatibility_projection_v1.py` the required bridge; that authority was then normalized again in a dated `2026-03-29` decision entry and only later reflected in the machine-readable authority pointer via `ACTIVE_RUN.json` on `2026-03-30` (`MDEC090`). In short: the switch was **incremental**, with design narrowing first, deterministic drift second, governance switch third, and `ACTIVE_RUN` repointing last.

# Section 3: Detailed reasoning

The transition did **not** happen all at once.

The earliest layer is a **contract change without authority change**. On `2026-03-06`, the repo explicitly assigned formulation-instance semantics to the LLM and separated deterministic responsibilities such as normalization and export. That appears in [docs/methods/llm_deterministic_responsibility_contract_2026-03-06.md](../methods/llm_deterministic_responsibility_contract_2026-03-06.md), in [docs/snapshots/snapshot_2026-03-06_llm_rule_audit_and_contract.md](snapshot_2026-03-06_llm_rule_audit_and_contract.md), and in [project/4_DECISIONS_LOG.md:87](../../project/4_DECISIONS_LOG.md#L87). That is not yet a move away from the LLM path; it is a narrowing-and-clarification of what the LLM should own.

The next layer is **practical drift inside the still-active LLM extractor era**. By `2026-03-13`, the project had already inserted deterministic logic into the Stage2 boundary for numbered DOE tables. The key evidence is [project/4_DECISIONS_LOG.md:909](../../project/4_DECISIONS_LOG.md#L909) and [data/mem/v1/dec.tsv:12](../../data/mem/v1/dec.tsv#L12), which say deterministic enumeration is primary and the LLM becomes “judge rather than row counter.” That is a real transfer of practical responsibility, but it still happens **inside** the legacy `auto_extract_weak_labels_v7pilot_r3_fixparse.py` mainline. Repo snapshots as late as March 18 still close DEV15 against a legacy-extractor-rooted benchmark lineage, so this drift did not yet amount to a mainline authority switch.

The third layer is **replacement planning before adoption**. The strongest evidence is the `2026-03-25` decision block added in commit `a4aeccd`. Those entries approve a semantic-object replacement direction, add a compatibility adapter, and validate both lift-based and true semantic replacement paths. But the language is explicit that this is not yet the mainline switch. The log says: “without switching the active runtime,” “Current benchmark-valid mainline remains Stage2 -> Stage3 -> Stage5 with the maintained wide-row Stage2 extractor,” and “the compatibility adapter exists... not to redefine the active benchmark runtime by itself.” This matters because it rules out a March 25 authority switch.

The strongest evidence for the **formal authority switch** is commit `75c148e` on `2026-03-26`. That commit simultaneously:

- changed [README.md](../../README.md) from naming `auto_extract_weak_labels_v7pilot_r3_fixparse.py` as the “Current active Stage 2 script” to naming `emit_semantic_objects_from_cleaned_papers_v1.py`
- added “Current active compatibility bridge” for `build_stage2_compatibility_projection_v1.py`
- demoted the legacy extractor to “Deprecated legacy Stage 2 fallback”
- rewrote [project/2_ARCHITECTURE.md](../../project/2_ARCHITECTURE.md), [project/ACTIVE_PIPELINE_FLOW.md](../../project/ACTIVE_PIPELINE_FLOW.md), [project/ACTIVE_PIPELINE_RUNBOOK.md](../../project/ACTIVE_PIPELINE_RUNBOOK.md), and [project/PIPELINE_SCRIPT_MAP.md](../../project/PIPELINE_SCRIPT_MAP.md) so that the active mainline becomes semantic Stage2 -> compatibility adapter -> Stage3 -> Stage5
- inserted `MDEC076` into [data/mem/v1/dec.tsv](../../data/mem/v1/dec.tsv), formally recording the migration

So, as a matter of repository history, the mainline wording switch shows up in the March 26 commit boundary. However, the decision text now preserved in [project/4_DECISIONS_LOG.md:2514](../../project/4_DECISIONS_LOG.md#L2514) is dated under the `2026-03-29` heading, and commit `03e5d25` adds a normalization note that older mentions of the “active Stage2 extractor” are pre-migration context only. That means the repo preserves both a **commit-time switch** (`2026-03-26`) and a **dated governance entry** (`2026-03-29`) for the same authority state.

Finally, there is a fourth layer: **machine-readable authority promotion lagged behind doc/memory authority**. [data/mem/v1/dec.tsv:31](../../data/mem/v1/dec.tsv#L31) explicitly says that governed docs and code already declared semantic Stage2 the active mainline, but `ACTIVE_RUN.json` still pointed to a historical March 14 legacy-extractor lineage. The current [data/results/ACTIVE_RUN.json](../../data/results/ACTIVE_RUN.json#L1) shows the semantic lineage and says `updated_at: "2026-03-30T00:00:00Z"`. So the repo stopped *describing* the LLM wide-row extractor as active before it stopped *default-resolving* to a legacy-extractor run.

Fact vs inference:

- **Fact:** March 6 narrowed and clarified the LLM/deterministic split.
- **Fact:** March 13 inserted deterministic row enumeration as primary for strong DOE tables.
- **Fact:** March 25 approved replacement direction but explicitly kept the legacy extractor as active runtime.
- **Fact:** March 26 commit `75c148e` rewrote active pipeline docs and memory to semantic Stage2 authority.
- **Fact:** March 29 decision log entry and `MDEC076` state that semantic Stage2 authority migrated.
- **Fact:** March 30 `MDEC090` says `ACTIVE_RUN` was promoted only after docs/code already switched.
- **Inference:** The switch was incremental and only later normalized, not a single moment with one authoritative artifact update.

# Section 4: Open uncertainties

The remaining ambiguity is **not** whether the switch happened; it clearly did. The ambiguity is about the exact dating layer to privilege:

- The earliest **commit-level governance rewrite** is `75c148e` on `2026-03-26`.
- The dated **decision-log wording** that states the migration sits under `2026-03-29` in [project/4_DECISIONS_LOG.md](../../project/4_DECISIONS_LOG.md).
- The **machine-readable active-run pointer** was repointed on `2026-03-30`, not at the time of the governance switch.

One additional uncertainty is git traceability of [data/results/ACTIVE_RUN.json](../../data/results/ACTIVE_RUN.json). In the current working tree the file exists, but `git ls-files --error-unmatch data/results/ACTIVE_RUN.json` reports that it is not known to git, and `git log --follow -- data/results/ACTIVE_RUN.json` returns no history. So the repo evidence proves the current contents and the memory/decision trail around its repointing, but it does **not** let us name a git commit that introduced the file’s current contents. The cleanest remaining settlement artifact would be a tracked commit for `data/results/ACTIVE_RUN.json` or a reproducibility-grade run/promotion note explicitly checked into git at the moment the pointer changed.
