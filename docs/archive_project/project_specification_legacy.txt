## Development Philosophy

This project follows a step-by-step debugging and verification workflow to ensure full reproducibility and transparency at every stage of the data extraction and processing pipeline.

Key Principles:

1. Incremental Debugging:
   Each processing step (e.g., CSV → Manifest → Clean → Weak Label → Ground Truth Template) is developed, debugged, and validated individually before proceeding to the next stage.

2. Standalone Executability:
   Every Python script must be independently executable from the command line, with all required arguments defined in its own argparse section.
   No script should rely on patch files, ad-hoc code snippets, or manual editing between runs.

3. Direct Replace Policy:
   When a bug is found, the preferred solution is to replace the entire file with a clean, fully corrected version, rather than applying small inline patches.
   This guarantees that each saved .py file is a ready-to-run version reflecting the latest stable state.

4. Minimal Reproduction Commands:
   Each script includes at the top a short example command showing the minimal reproducible use case, designed for quick verification (e.g., processing 3–5 sample rows).

5. Launch.json Integration:
   For local debugging in VS Code, every new or modified script has a corresponding launch.json configuration, enabling one-click execution in an isolated, traceable environment.

6. No One-Shot Full Runs:
   Full-dataset execution is only performed after all intermediate steps have been individually validated. This saves time, tokens, and debugging overhead.
   Explicit criteria for when to scale up from small-batch to full-dataset runs.

## Formulation Reconstruction Pipeline

The architecture after document cleaning extends beyond direct field filling and reconstructs full formulation records through staged processing:

Document preprocessing
-> semantic extraction (LLM)
-> formulation hypothesis layer
-> evidence binding
-> formulation assembly
-> formulation-level audit
-> final tabular export

Role separation:

LLM responsibilities:
- semantic understanding of scientific text
- formulation instance detection
- interpretation of inheritance or similarity descriptions

Rule-based system responsibilities:
- deterministic evidence binding
- numeric/token matching
- normalization
- verification and audit preparation

## Error Recovery Rules

SystemExit: 2
→ Missing args; check manifest path.

No HTML/PDF found
→ Verify paths in manifest_current.tsv and confirm files exist on disk.

No extractable text (scanned/encrypted?)
→ Run OCR first or skip.

Unrecognized arguments
→ Legacy CLI call; switch to manifest mode or patch build_arg_parser.

## Goal

Build a clean, auditable, and reproducible formulation database extracted from scientific literature.

The final dataset is a tabular formulation dataset where each row represents a formulation and each column represents a structured parameter.

The database is designed to support data-driven analysis of formulation design, including statistical analysis, predictive modeling, and discovery of empirical formulation patterns.

The pipeline may use richer intermediate representations internally, but the final output always remains a tabular dataset suitable for downstream scientific analysis.

## Internal Representation vs Final Output

Scientific papers often describe formulations with distributed evidence across narrative text, tables, and captions. Because this evidence is fragmented, the pipeline reconstructs formulation records through intermediate stages instead of directly mapping local text spans to final table rows.

Conceptual intermediate stages:
- semantic extraction of formulation information
- formulation hypothesis generation
- evidence binding to document spans or table cells
- formulation-level record assembly
- formulation-level audit

After verification and audit checks, reconstructed formulation records are flattened into the final tabular dataset for downstream analysis and modeling.

## Architecture Summary

The system evolved from a field-first extraction design to a formulation-level reconstruction pipeline.

The pipeline reconstructs formulation records from distributed evidence before exporting them as a structured table.
